Section 01
Overview & Introduction

Managing TLS certificates at scale inside a national school network is a recurring operational challenge. Thousands of internal services — learning management platforms, authentication portals, DNS servers, and student devices — must communicate securely without exposing a public Certificate Authority or paying per-certificate fees.

This project describes a lightweight, self-hosted PKI built on Smallstep Certificate Authority and the ACME protocol, capable of issuing, renewing, and revoking X.509 certificates automatically across the entire Algeria RESEN/ENS network.

■ Objectives

  • Eliminate public CA dependency for internal services
  • Automate the full certificate lifecycle end-to-end
  • Scale to 500+ endpoints across Algeria
  • Enforce Zero-Trust on all school services
Requirement Specification Notes
OS WSL2 / Ubuntu 22.04 WSL2 preferred for networking
CA Software step-ca Open-source, RFC 8555
ACME Clients certbot / acme.sh Any RFC 8555 compatible
Port 9000 (HTTPS) Firewall must allow inbound
Root Key Storage YubiKey 5 / HSM Never on networked machine
Monitoring Prometheus + Grafana step_ca_x509_* metrics
Section 02
Relationship with Cybersecurity

PKI is one of the cornerstones of modern cybersecurity, providing the cryptographic foundations for the four pillars of the CIA+ model — Confidentiality, Integrity, Authentication, and Non-repudiation.

🔒
Confidentiality
Data accessible only to authorized parties, preventing unauthorized disclosure.
TLS via X.509 Certificates
🛡️
Integrity
Data must remain unaltered in transit from sender to receiver.
Digital Signatures
🪪
Authentication
Entities must cryptographically prove their identity before gaining access.
mTLS — Mutual Cert Validation
📋
Non-repudiation
Actions cannot be denied after the fact — all operations are traceable.
Cert-bound Audit Logs
Zero Trust
Zero-Trust Architecture

In a Zero-Trust model, no device or service is trusted by default — every connection must present a valid, CA-signed certificate. A private PKI becomes the trust anchor for the entire network, replacing implicit network-level trust with explicit cryptographic identity.

01
Explicit Verification
Every request is authenticated via mTLS regardless of network location or origin. No implicit trust zones exist within the infrastructure.
02
Least-Privilege Access
Short-lived certificates with 24h default validity dramatically limit the blast radius of any single compromise — stale credentials expire automatically.
03
Assume Breach
Rapid revocation via CRL/OCSP neutralises compromised credentials within 60 seconds. The system is designed to recover instantly from key compromise.
Section 03
PKI — Public Key Infrastructure

A PKI is a system of roles, policies, hardware, software, and procedures needed to create, manage, distribute, store, and revoke digital certificates. At its heart sits an asymmetric key pair: a private key held by the entity and a public key embedded in a certificate signed by a trusted CA.

Core Components
Component Role Security Posture
Root CA Self-signed trust anchor Air-gapped
Intermediate CA Issues end-entity certificates Online
End-Entity Cert Identifies server, device, or user 24h–90d
CRL / OCSP Invalidate compromised certs Distributed
Certificate Profile Defines key usage, SAN, extensions Policy-enforced
Certificate Lifecycle
1
Enroll
Device generates key pair and submits a Certificate Signing Request (CSR) to the CA.
Client / ACME agent
2
Issue
CA validates identity through the selected challenge type and signs the certificate.
Smallstep CA
3
Deploy
Certificate is installed on the service; TLS handshake succeeds for all clients.
Server / application
4
Renew
ACME client automatically renews at 2/3 of the validity period without human intervention.
certbot / acme.sh
5
Revoke
Compromised cert is added to CRL and OCSP is updated — propagated within 60 seconds.
Admin / hook
Section 04
Smallstep CA — In Depth

Smallstep (step-ca) is an open-source, cloud-native Certificate Authority written in Go. It implements RFC 5280 (X.509) and RFC 8555 (ACME), supporting provisioners including JWK, OIDC, AWS/GCP/Azure metadata, and SSH certificates. A single binary replaces complex EJBCA or OpenSSL setups with a modern REST API.

⚡
ACME Server Built-In
Compatible with certbot, acme.sh, Caddy, and Traefik out of the box. No extra server required.
⏱️
Short-Lived Certificates
24-hour default validity dramatically reduces the revocation burden — certificates auto-expire.
🔄
Automated Renewal
step-ca renew daemon and ACME renewal information endpoint handle lifecycle automatically.
🔑
SSH Certificate Authority
Replaces SSH authorized_keys management with modern certificate-based authentication.
🔐
Hardware-Backed Root Keys
YubiKey 5 (PKCS#11) and TPM 2.0 support for root key protection — never software-only.
📊
Full JSON Audit Log
All issuance, renewal, and revocation events are logged and queryable via Prometheus/Grafana.
Internal Architecture

Smallstep runs as a single daemon process, storing state in an embedded BadgerDB (or external MySQL/PostgreSQL). It exposes two main endpoints: a HTTPS management API and an ACME directory endpoint, both on port 9000 by default.

Layer Component Description
Issuance Intermediate CA step-ca daemon — signs all leaf certificates daily
Automation ACME Provisioner RFC 8555 — http-01 / tls-alpn-01 challenges supported
Identity JWK Provisioner Token-based for VMs, CI/CD, non-HTTP services
Monitoring Audit Log + Prometheus Full observability — Grafana dashboards for metrics
Trust Anchor Root CA Self-signed; private key on YubiKey or air-gapped host
Section 05
ACME Protocol — In Depth

ACME (Automated Certificate Management Environment) is an IETF standard (RFC 8555) defining a machine-to-machine protocol that allows software to interact with a CA to automate the full certificate lifecycle — order, validate, issue, and renew — without any human intervention.

ACME Workflow
1
Account Registration
Client generates an account key pair and registers with the CA's ACME endpoint to establish an identity.
2
Order Creation
Client submits the domain name(s) for which a certificate is required, creating a certificate order.
3
Challenge Selection
CA returns available challenge types: http-01, dns-01, or tls-alpn-01 for domain control validation.
4
Domain Validation
Client proves control by serving a signed token at the challenge URL or DNS record as required.
5
Certificate Issuance
CA validates the proof, signs the CSR, and returns the DER-encoded certificate chain to the client.
6
Automatic Renewal
Client polls /renewalInfo or monitors expiry; automatically renews before the deadline with no manual intervention.
Challenge Types
http-01
Serve a token at /.well-known/acme-challenge/ on port 80. The CA fetches and validates the token over HTTP.
Web Servers · LMS Portals · Dashboards
tls-alpn-01
Serve the token in a TLS handshake via the ALPN extension on port 443. No HTTP port required.
Services where HTTP port unavailable
dns-01
Create a TXT record in the authoritative DNS zone. Required for wildcard certificates.
Wildcard certs for *.school.dz
Section 06
Architecture — National School Network

Algeria's national school network (RESEN / ENS model) connects hundreds of establishments through a central ministry network. The PKI is positioned at the national datacenter, with per-wilaya edge nodes acting as Registration Authorities (RA) that cache OCSP responses and reduce latency for local schools.

Tier Location Role Components
Tier 0 — Root Air-gapped vault Trust Anchor HSM / YubiKey + offline Root CA
Tier 1 — Issuing CA National Datacenter Certificate Issuance step-ca + ACME provisioner
Tier 2 — RA Nodes Wilaya (regional) servers Proximity Enrolment step-ca RA + OCSP cache
Tier 3 — Endpoints Each school server/device Certificate Consumers certbot / acme.sh / Traefik
Tier 4 — Clients Student and staff devices TLS + mTLS Verification OS trust store + CA bundle via MDM
🏛️
■ Trust Distribution
The root CA certificate is distributed to all school machines via the Ministry's MDM (Mobile Device Management) and Group Policy (GPO), ensuring every device trusts the internal certificate chain without browser security warnings. This centralised distribution ensures uniform trust across the entire RESEN/ENS network.
Section 07
Step-by-Step Deployment
1
Environment Setup
$ wsl --install
$ wsl --set-default-version 2
$ sudo apt update && sudo apt upgrade -y
Enable WSL2 with Ubuntu 22.04 LTS as the default distribution. Set WSL2 as default for proper Linux kernel and network compatibility required for step-ca port binding. Update all packages before installation.
2
Install step CLI & CA Server
$ curl -fsSL https://raw.githubusercontent.com/
smallstep/cli/master/install.sh | bash
$ curl -fsSL https://raw.githubusercontent.com/
smallstep/certificates/master/install.sh | bash
$ step version && step-ca version
Download and install both the step CLI (primary client for bootstrapping and managing provisioners) and the step-ca daemon binary (the CA server that listens for and signs certificate requests).
3
Initialize the CA
$ step ca init
$ step path
$ # Generates root CA, intermediate CA, ca.json
Launch the interactive wizard generating the root CA certificate, intermediate CA certificate, and ca.json configuration. Key prompts: deployment type (Standalone), CA name, DNS/IP, port (:9000), provisioner email, and encryption password.
4
Add ACME Provisioner
$ step ca provisioner add acme --type ACME
$ step ca provisioner list
Register an RFC 8555 ACME provisioner. Once added, any ACME-compatible client (Certbot, acme.sh, Caddy) can automatically request and renew certificates via http-01 or tls-alpn-01 challenges without human intervention.
5
Run the CA Server
$ step-ca $(step path)/config/ca.json
$ # Or as a persistent systemd service:
$ sudo systemctl enable step-ca && sudo systemctl start step-ca
Start the CA server. For production deployments, register as a systemd service for automatic startup on boot. The CA listens on https://localhost:9000 by default.
6
Bootstrap Clients & Issue Certs
$ step certificate fingerprint $(step path)/certs/root_ca.crt
$ step ca bootstrap --ca-url https://localhost:9000 --fingerprint <FP>
$ step ca certificate myservice.local cert.crt cert.key
Get the SHA-256 fingerprint of the root CA, bootstrap every client machine to trust the internal CA, then issue certificates manually or via ACME clients like Certbot for automated lifecycle management.
Section 08
Problems Treated & Solutions
P1
Browser Certificate Warnings
Problem
Browsers display 'Not Secure' when internal sites use an unknown CA that isn't in the system trust store.
Solution
Distribute the root CA via MDM/GPO. On Linux: update-ca-certificates. On Windows: certutil or Group Policy. Use step ca bootstrap for one-command trust installation.
P2
Certificate Expiry in Production
Problem
Short-lived certificates (24h) expire if the ACME renewal daemon stops, causing service outages.
Solution
Run certbot renew via cron every 12 hours. Use step ca renew --daemon for background renewal. Monitor with Prometheus and alert at 6-hour remaining validity.
P3
Scaling to 500+ Endpoints
Problem
A single CA node becomes a bottleneck and single point of failure at the national scale with hundreds of schools.
Solution
Deploy Regional RA nodes per wilaya. RA nodes forward requests to the central CA, cache OCSP responses, and serve CRL locally — reducing latency and CA load by approximately 70%.
P4
Wildcard Certificate Automation
Problem
The http-01 challenge cannot issue wildcard certificates (*.school.dz), blocking the entire subdomain.
Solution
Use dns-01 challenge with an internal ACME DNS solver. Deploy acme-dns or a custom hook updating the internal BIND/PowerDNS zone. Integrate with certbot --preferred-challenges dns-01.
P5
Compromised Private Key
Problem
A server private key is leaked — all communications on that service are at immediate risk of interception.
Solution
Immediately revoke with step ca revoke <serial>. Re-issue with a freshly generated key pair. CRL/OCSP propagates revocation to all RA nodes within 60 seconds.
P6
Air-Gap Root CA Management
Problem
The root CA private key must never touch a network-connected machine, yet needs to be usable for ceremonies.
Solution
Bootstrap root CA offline on an air-gapped machine with a YubiKey 5 (PKCS#11). Sign the intermediate CA certificate once during the key ceremony. Store the root key in a hardware vault. All daily operations use only the intermediate CA.
